# 2026-04-24: Newsletter List Inline Previews + Hover Tooltip

## Summary

Added Gmail-style email body previews inside the newsletter list: each row now shows a short muted snippet after the subject, and a hover tooltip reveals the first few lines after a sustained 2-second hover. Previews are generated client-side at runtime — the `data/index.json` manifest stays untouched. Both the newsletter listing page and bookmarks page benefit, since they share `renderEmailList`.

## Changes

- **templates/app.js**: Added a new "Email Body Previews" section with `PreviewCache` (in-memory `Map` keyed by file path), `fetchPreview` (throttled to 4 concurrent requests, deduplicated via in-flight map), `extractPreviewText` (DOMParser-based — strips `script`/`style`/`head`/hidden nodes, collapses whitespace, preserves block boundaries as line breaks), `initPreviewObserver` (`IntersectionObserver` with `200px` rootMargin, populates `short` snippet on first viewport entry), and `initPreviewTooltip` (delegated `mouseover`/`mouseout` on the list container with `relatedTarget` containment checks; 2000 ms `setTimeout`; shared tooltip element positioned below the hovered row, flipped above when out of room). Extended the `renderEmailList` row template with `<span class="email-item__preview" aria-hidden="true"></span>` and wired the two init calls after `bindListActions`.
- **templates/style.css**: Added `.email-item__preview` (flex: 1, muted colour, single-line ellipsis, `::before` dash separator gated by `:not(:empty)`), tightened `.email-item__subject` to single-line ellipsis at `max-width: 55%` so the preview sits beside it rather than below, and added a new "Hover Preview Tooltip" block with `.preview-tooltip` (absolutely positioned, `pre-wrap`, fade-in via `data-visible`, `prefers-reduced-motion` respected).
- **ARCHITECTURE.md**: Added a new "Email Body Previews (Lazy Fetch)" subsection under "Client-Side Architecture" describing the flow, concurrency cap, and caching model.

## Key Design Choices

- **No build changes**: All extraction happens in the browser — kept the "build once, render client-side" principle intact and avoided bloating the 3.1 MB manifest.
- **Single fetch, two forms**: one HTTP request per email yields both the short inline snippet (160 chars) and the long tooltip excerpt (500 chars). The HTTP layer + in-memory `Map` together handle reuse — no sessionStorage needed.
- **Delegated tooltip listeners**: one pair of `mouseover`/`mouseout` handlers on the list container (not per-row) survive re-renders from search filtering without rebinding.
- **CSS tokens only**: preview + tooltip use `--fg-muted`, `--fg-faint`, `--bg-surface`, `--border-strong`, so the three-state theme toggle works automatically in both inline and hover states.
